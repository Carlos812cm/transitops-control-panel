# TransitOps Platform v1.5.0: API Pagination & Response Metadata

## Release summary

v1.5.0 focuses on improving the professional quality of the TransitOps Back-End API by adding pagination support and response metadata to list endpoints.

This release prepares the API for larger datasets while preserving compatibility with the existing Angular Front-End.

## Objective

Add pagination and response metadata to core API list endpoints without breaking current consumers.

The existing Front-End should continue working with `response.data`, while new API responses expose a `meta` object that can be used later to build paginated UI controls.

## Scope

### Included

- Add reusable Back-End pagination utilities.
- Extend successful API responses with optional metadata.
- Add `page`, `limit` and `pageSize` support to list endpoint query schemas.
- Apply pagination to:
  - Vehicles
  - Drivers
  - Transit Routes
  - Trips
- Preserve existing filters and search behavior.
- Preserve existing list endpoint URLs.
- Add Back-End tests for pagination behavior.
- Update Front-End API response typings to support optional metadata.
- Update Front-End query parameter typings to align with API pagination.
- Document QA results for the release.

### Out of scope

- No Angular pagination UI.
- No visual table or card layout changes.
- No API route renaming.
- No Prisma schema changes.
- No database migrations.
- No authentication logic changes.
- No role or permission changes.
- No deployment configuration changes.

## Problem statement

Current list endpoints return all matching records in a single response.

This works for small datasets, but it does not scale well and does not expose useful metadata for professional API consumers.

Examples of current behavior:

- `GET /api/vehicles` returns all vehicles.
- `GET /api/drivers` returns all drivers.
- `GET /api/routes` returns all transit routes.
- `GET /api/trips` returns all trips.

A professional API should allow clients to request specific pages and understand the total number of records available.

## Target API behavior

### Supported query parameters

```text
page=1
limit=10
pageSize=10
```

### Compatibility rule

Both limit and pageSize should be accepted.
If both are provided, limit should take precedence.

### Default pagination

If no pagination parameters are provided:

```text
page=1
limit=10
```

### Maximum page size

The API should prevent very large requests by enforcing a maximum limit.
Recommended maximum:

```text
limit=100
```

### Target response format

```json
{
  "success": true,
  "message": "Vehicles retrieved successfully.",
  "data": [],
  "meta": {
    "page": 1,
    "limit": 10,
    "total": 42,
    "totalPages": 5,
    "hasNextPage": true,
    "hasPreviousPage": false
  }
}
```

## Technical approach

### Shared pagination utilities

Create shared utilities under:
`server/src/common/pagination/`

Expected responsibilities:

- Normalize query values.
- Coerce page and limit values safely.
- Calculate Prisma skip and take.
- Build response metadata.
- Provide reusable TypeScript interfaces.

### API response extension

Update:
`server/src/common/responses/api-response.ts`

Add optional metadata support:

```typescript
meta?: PaginationMeta
```

`meta` must remain optional to preserve compatibility with non-list endpoints.

### Service layer changes

Each list service should return a paginated result instead of a raw array.

Expected shape:

```typescript
{
  data: T[];
  meta: PaginationMeta;
}
```

### Prisma query strategy

Each paginated list should use:

```typescript
prisma.$transaction([
  prisma.model.count({ where }),
  prisma.model.findMany({
    where,
    skip,
    take,
    orderBy
  })
])
```

This keeps the count and page query aligned.

### Controller layer changes

Each list controller should send:

```typescript
return sendSuccess(response, 'Message.', result.data, 200, result.meta);
```

### Front-End typing changes

Update:

- `src/app/core/models/api-response.model.ts`
- `src/app/core/models/query-params.model.ts`

The Front-End should support meta but should not require it.

## Risk assessment

### Low risk
- Extending API responses with optional metadata.
- Updating Front-End typings without changing UI behavior.
- Adding reusable pagination helpers.

### Medium risk
- Tests may assume list endpoints return all records.
- Front-End components may rely on unpaginated responses for complete local filtering.
- Trips include related entities, so pagination must preserve includes.

### Mitigation
- Keep data as an array.
- Keep response success/message/data contract unchanged.
- Add meta as optional.
- Use default limit that keeps current demo usable.
- Update tests intentionally.
- Run full local quality gate before opening PR.
- Let GitHub Actions validate the PR automatically.

This release is complete when:

- List endpoints accept page.
- List endpoints accept limit.
- List endpoints accept pageSize.
- limit takes precedence over pageSize.
- Responses include meta.
- meta.total reflects filtered total records.

- meta.hasNextPage is accurate.
