# TransitOps Platform v1.3.0: Responsive Navbar & User Menu Polish

## Release summary

v1.3.0 focuses on improving the responsive behavior of the authenticated navbar experience in TransitOps Control Panel.

This release refines the user profile trigger in mobile and tablet layouts, reduces visual clutter in the navbar, and stabilizes the sticky navbar behavior while preserving the existing desktop experience.

## Objective

Improve the responsive navbar and authenticated user menu behavior without changing application logic, authentication, API contracts, Prisma schema, or Back-End functionality.

## Scope

### Included

- Keep the full user profile trigger on desktop:
  - Avatar
  - Display name
  - Role
  - Chevron
- Use a compact avatar-only profile trigger on responsive layouts.
- Preserve the existing user dropdown behavior:
  - Open on avatar/profile trigger click.
  - Close on outside click.
  - Close on Escape.
  - Navigate to Profile.
  - Navigate to Settings.
  - Logout.
- Improve dropdown alignment on smaller screens.
- Prevent the responsive navbar from growing unnecessarily.
- Stabilize sticky navbar behavior during scroll.
- Ensure compatibility with the responsive sidebar and mobile sidebar toggle.
- Validate behavior in light and dark themes.
- Document QA results before release closure.

### Out of scope

- No Back-End changes.
- No Prisma schema changes.
- No database migrations.
- No API endpoint changes.
- No authentication logic changes.
- No role/permission changes.
- No language service contract changes.
- No new third-party UI libraries.
- No deployment configuration changes.

## Problem statement

### Problem 1: Responsive user block is too large

In responsive mode, the navbar currently displays the complete authenticated user block as a full-width card-like element.

Current responsive behavior shows:
- Avatar
- User name
- Role
- Chevron

On mobile and tablet layouts, this consumes too much navbar space and makes the header feel visually heavy.

Expected behavior:
- Mobile/tablet should show only the user avatar/logo.
- Clicking or tapping the avatar should open the same profile dropdown.
- Desktop should keep the current complete profile trigger.

### Problem 2: Navbar sticky behavior needs refinement on responsive layouts

The navbar should remain visually stable while scrolling down in responsive layouts.

Expected behavior:
- Navbar remains sticky/fixed according to the current layout architecture.
- Mobile sidebar toggle remains usable.
- Sidebar overlay/backdrop behavior remains compatible.
- No visual overlap between navbar, sidebar toggle, dropdown and page content.
- No layout jumps during scroll.

## Technical approach

### Strategy

Use the current navbar component and refine its responsive behavior primarily through SCSS.
Avoid duplicating profile buttons or creating a separate mobile-only component unless strictly necessary.

### Preferred implementation

- Keep the existing .profile-trigger as the single source of interaction.
- Hide .user-copy and .profile-chevron in responsive layouts.
- Keep .user-avatar visible and clickable.
- Adjust .navbar-panel responsive layout to remain compact.
- Rework mobile dropdown width/alignment to avoid full-width card behavior.
- Preserve the current Angular signal-based dropdown state.
- Avoid TypeScript changes unless needed for accessibility.

## Risk assessment

### Low risk
- SCSS-only responsive refinements.
- Preserving current component structure.
- Preserving existing dropdown logic.

### Medium risk
- Sticky behavior may depend on parent container layout.
- Mobile sidebar toggle uses fixed positioning and high z-index.
- Dropdown alignment may need special handling near viewport edges.

### Mitigation
- Prefer minimal CSS changes.
- Avoid modifying layout shell unless required.
- Test across mobile, tablet and desktop sizes.
- Validate with sidebar open and closed.
- Validate scroll behavior on long pages.
