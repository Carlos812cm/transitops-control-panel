# TransitOps Platform v1.3.0: Responsive Navbar & User Menu Polish

## Release summary

v1.3.0 focuses on improving the responsive behavior of the authenticated navbar experience in TransitOps Control Panel.

This release refines the user profile trigger in mobile and tablet layouts, reduces visual clutter in the navbar, and stabilizes the navbar behavior while preserving the existing desktop experience.

## Objective

Improve the responsive navbar and authenticated user menu behavior without changing application logic, authentication, API contracts, Prisma schema, or Back-End functionality.

## Scope

### Included

- Keep the full user profile trigger on desktop: avatar, display name, role and chevron.
- Use a compact avatar-only profile trigger on responsive layouts.
- Preserve the existing user dropdown behavior: open, close, Profile, Settings and Logout.
- Improve dropdown alignment on smaller screens.
- Prevent the responsive navbar from growing unnecessarily.
- Stabilize fixed navbar behavior during scroll.
- Ensure compatibility with the responsive sidebar and mobile sidebar toggle.
- Validate behavior in light and dark themes.
- Document QA results before release closure.

### Out of scope

- No Back-End changes.
- No Prisma schema changes.
- No database migrations.
- No API endpoint changes.
- No authentication logic changes.
- No role or permission changes.
- No language service contract changes.
- No new third-party UI libraries.
- No deployment configuration changes.

## Problem statement

### Problem 1: Responsive user block is too large

In responsive mode, the navbar previously displayed the complete authenticated user block as a full-width card-like element.

Expected behavior:

- Mobile and tablet layouts show only the user avatar/logo.
- Clicking or tapping the avatar opens the same profile dropdown.
- Desktop keeps the current complete profile trigger.

### Problem 2: Navbar scroll behavior needed refinement on responsive layouts

The navbar should remain visually stable while scrolling down in responsive layouts.

Expected behavior:

- Navbar remains visible according to the current layout architecture.
- Mobile sidebar toggle remains usable.
- Sidebar overlay and backdrop behavior remains compatible.
- No visual overlap between navbar, sidebar toggle, dropdown and page content.
- No layout jumps during scroll.

## Technical approach

### Strategy

Use the current navbar component and refine its responsive behavior primarily through SCSS.
Avoid duplicating profile buttons or creating a separate mobile-only component unless strictly necessary.

### Implementation summary

- Kept the existing profile trigger as the single source of interaction.
- Hid user copy and chevron in responsive layouts.
- Kept the user avatar visible and clickable.
- Adjusted the responsive navbar to remain compact.
- Reworked mobile dropdown width and alignment to avoid full-width card behavior.
- Preserved the Angular signal-based dropdown state.
- Added an accessible label to the profile trigger.
- Added responsive layout spacing so page content does not render underneath the fixed navbar.
- Restored compact mobile main content padding.

## Risk assessment

### Low risk

- SCSS-only responsive refinements.
- Preserving current component structure.
- Preserving existing dropdown logic.

### Medium risk

- Fixed responsive navbar behavior depends on parent container layout.
- Mobile sidebar toggle uses fixed positioning and high z-index.
- Dropdown alignment may need special handling near viewport edges.

### Mitigation

- Prefer minimal CSS changes.
- Avoid modifying layout shell unless required.
- Test across mobile, tablet and desktop sizes.
- Validate with sidebar open and closed.
- Validate scroll behavior on long pages.
