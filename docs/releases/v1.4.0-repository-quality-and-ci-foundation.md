# TransitOps Platform v1.4.0: Repository Quality & CI Foundation

## Release summary

v1.4.0 focuses on improving repository quality, automated validation and release discipline for TransitOps Platform.

This release does not introduce new product features. Its goal is to make the project more professional, easier to validate and safer to evolve through GitHub Actions and cleaner project metadata.

## Objective

Add a CI foundation and repository quality improvements without changing application behavior, UI flows, API contracts, Prisma schema or database functionality.

## Scope

### Included

- Add a GitHub Actions CI workflow.
- Run the main quality gate automatically on pull requests and pushes to main.
- Validate Angular build.
- Validate mock API tests.
- Validate Back-End typecheck.
- Validate Back-End tests.
- Validate Back-End build.
- Clean duplicated or misplaced scripts from the Back-End package file.
- Align project version metadata with the release version.
- Document the v1.4.0 quality strategy.

### Out of scope

- No UI feature changes.
- No layout changes.
- No API endpoint changes.
- No Prisma schema changes.
- No database migrations.
- No authentication logic changes.
- No role or permission changes.
- No deployment provider configuration.
- No production hosting setup.

## Problem statement

The project already has a strong manual quality gate, but the repository does not yet validate changes automatically through GitHub Actions.

Current validation depends on manually running:

- npm run build
- npm run test:api
- npm run api:real:typecheck
- npm run api:real:test
- npm run api:real:build

This is acceptable for local development, but a professional repository should also validate pull requests automatically before merging into main.

## Technical approach

### Strategy

Add a minimal but professional GitHub Actions workflow that validates the same commands already used during local releases.

The first CI version should be simple, readable and stable. Complexity should be added only if the Back-End tests require services such as PostgreSQL inside the workflow.

### Preferred implementation

- Create `.github/workflows/ci.yml`.
- Trigger CI on pull requests targeting main.
- Trigger CI on pushes to main.
- Use Node.js setup.
- Install root dependencies with `npm ci`.
- Run Angular and mock API checks from the root.
- Install Back-End dependencies inside `server`.
- Run Back-End typecheck, tests and build.
- Keep CI aligned with the documented local quality gate.

## Risk assessment

### Low risk

- Documentation-only release planning.
- CI workflow uses existing project scripts.
- Repository metadata changes do not affect runtime behavior.

### Medium risk

- Back-End tests may require environment variables in CI.
- Back-End tests may require PostgreSQL service availability.
- Package lock files may change after dependency installation.

### Mitigation

- Start with a simple CI workflow.
- Run the quality gate locally before pushing.
- If CI fails because of missing database or environment variables, add only the required CI service configuration.
- Keep implementation commits small and reviewable.

## Acceptance criteria

This release is complete when:

- CI workflow exists under `.github/workflows`.
- CI runs on pull requests targeting main.
- CI runs on pushes to main.
- Angular build passes.
- Mock API tests pass.
- Back-End typecheck passes.
- Back-End tests pass.
- Back-End build passes.
- Package metadata is consistent.
- Back-End scripts are cleaned without removing required commands.
- PR is merged into main.
- Tag v1.4.0 is created.
